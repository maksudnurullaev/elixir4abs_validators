В Узбекистане структура 20-значного банковского счета (так называемого МФО-счета) строго регламентирована Центральным банком. Каждый сегмент номера несет определенную информацию.

Вот как расшифровывается структура счета:

### Структура 20-значного счета: `12345 678 9 12345678901`

|**Разряды**|**Наименование**|**Описание**|
|---|---|---|
|**1–5**|**Балансовый счет**|Указывает на тип счета (например, `20208` — для физлиц до востребования, `23120` — для суммовых карт).|
|**6–8**|**Код валюты**|ISO-код валюты. Например: `000` — сум (UZS), `840` — доллар США (USD).|
|**9**|**Контрольный ключ**|Вычисляется по специальному алгоритму для проверки корректности всего номера.|
|**10–20**|**Уникальный номер**|Порядковый номер счета клиента внутри конкретного филиала банка.|

---

### Основные типы счетов (первые 5 цифр):

- **20208** — Депозиты до востребования физических лиц (обычно привязаны к картам).
- **204xx / 206xx** — Сберегательные и срочные вклады граждан.
- **226xx** — Целевые средства (например, для покупки автомобиля или оплаты контракта).
- **23120** — Транзитные счета для операций с пластиковыми картами.

# Контрольный ключ

## Метод - 1

Алгоритм вычисления контрольного ключа (9-й разряд) в банковских счетах Узбекистана основан на методе **весовых коэффициентов**. Этот механизм позволяет системе моментально определить, не допустил ли операционист или клиент ошибку при вводе цифр.

Процесс происходит в несколько этапов:
### 1. Подготовка данных

Для расчета берется 20-значный номер счета, где на месте 9-го разряда (ключа) временно ставится **0**. Также в расчете участвует 5-значный код **МФО** (код филиала банка).

Итоговая последовательность для расчета состоит из 25 цифр:

`МММММ` (МФО) + `БББББ` (Баланс) + `ВВВ` (Валюта) + `0` (Ключ) + `XXXXXXXXXXX` (Лицевой счет).

### 2. Применение весовых коэффициентов

Каждая цифра этой 25-значной последовательности умножается на определенный «вес» из фиксированной таблицы. В Узбекистане используется следующий ряд коэффициентов (повторяющийся цикл):

**7, 1, 3, 7, 1, 3, 7, 1, 3, 7, 1, 3, 7, 1, 3...** и так далее до конца.

### 3. Суммирование произведений

После умножения каждой цифры счета на её вес, берется **последняя цифра** каждого полученного произведения. Эти цифры складываются между собой.

> **Пример:** Если 1-я цифра счета `2` и её вес `7`, то $2 \times 7 = 14$. Для суммы берем только `4`.

### 4. Финальный расчет ключа

Полученная общая сумма (пусть это будет $S$) снова сокращается до последней цифры (берется остаток от деления на 10).

Затем выполняется операция:

$$(S \times 3) \pmod{10}$$

Результат этого вычисления и есть та самая цифра, которая должна стоять на 9-м месте в номере счета.

## Метод - 2

Этот документ описывает другой, более специфический вариант вычисления контрольного ключа (K14), который используется в межбанковских клиринговых системах ЦБ Узбекистана.

В отличие от стандартного метода «весов», здесь применяется алгоритм **деления по модулю 11**.

Вот пошаговая расшифровка алгоритма из вашего изображения:

### 1. Формирование последовательности

Берется 25-значное число, состоящее из:
- **B1–B5**: Код банка (МФО).
- **C6–C10**: Балансовый счет.
- **V11–V13**: Код валюты.
- **S15–S22**: Уникальный регистрационный номер клиента (из базы НИББД).
- **N23–N25**: Порядковый номер счета.

### 2. Расчет суммы (SYMMA)

В этом алгоритме каждая цифра умножается на **следующую за ней** цифру в последовательности, и все эти произведения складываются:

$$(B1 \times B2) + (B2 \times B3) + \dots + (N24 \times N25) + (N25 \times 9)$$

### 3. Вычисление остатка

Полученная сумма делится на **11**, и определяется остаток от деления (**A**):

$$SYMMA \pmod{11} = A$$

### 4. Определение ключа (K)

Основная формула:
$$11 - A = K$$
**Исключения (важно):**
1. Если остаток **A = 0** (сумма делится на 11 без остатка), то ключ **K = 9**.
2. Если остаток **A = 1**, то ключ **K = 0**.

### Практическое применение

Этот алгоритм критически важен для вашей работы над проектом **Elixir4ABS**, так как он обеспечивает корректную идентификацию счетов в клиринговой системе.

Поскольку вы используете Elixir, реализация этого алгоритма через рекурсию или `Enum.reduce` будет выглядеть очень изящно. Учитывая специфику ваших интересов в области банковских систем (ABS) и архитектуры данных, правильная обработка этих 20-значных номеров — это фундамент всей логики транзакций.

Для реализации этого алгоритма на Elixir лучше всего подойдет `Enum.reduce` или `Enum.zip_with`, чтобы элегантно обработать пары соседних цифр.

Ниже представлен готовый модуль для вашего проекта **Elixir4ABS**, который учитывает структуру из документа и специфические правила для остатков 0 и 1.

Elixir

```elixir
defmodule Elixir4ABS.AccountValidator do
  @doc """
  Вычисляет контрольный ключ K14 на основе 25-значной последовательности.
  Ожидает список цифр [b1, b2, ..., n25].
  """
  def calculate_k14(digits) when length(digits) == 25 do
    # 1. Вычисляем сумму произведений соседних пар: (B1*B2) + (B2*B3) + ...
    # Enum.chunk_every(digits, 2, 1, :discard) создает пары [B1, B2], [B2, B3] и т.д.
    pairs_sum = 
      digits
      |> Enum.chunk_every(2, 1, :discard)
      |> Enum.map(fn [a, b] -> a * b end)
      |> Enum.sum()

    # 2. Добавляем последний член (N25 * 9) согласно формуле из документа
    last_digit = List.last(digits)
    total_sum = pairs_sum + (last_digit * 9)

    # 3. Находим остаток от деления на 11
    remainder = rem(total_sum, 11)

    # 4. Применяем логику исключений и финальный расчет
    cond do
      remainder == 0 -> 9
      remainder == 1 -> 0
      true -> 11 - remainder
    end
  end
end
```

### Как это работает согласно вашему документу:

- **Сдвиг по парам:** Мы используем `Enum.chunk_every(digits, 2, 1, :discard)`, что позволяет пройтись по списку как $(B1 \times B2) + (B2 \times B3) + \dots$ без ручного управления индексами.
- **Исключение 0:** Если `remainder == 0`, функция возвращает **9**.
- **Исключение 1:** Если `remainder == 1`, функция возвращает **0**.
- **Основной расчет:** В остальных случаях возвращается результат $11 - A$.

### Что стоит учесть при интеграции в ABS:

1. **Типы данных:** Убедитесь, что на вход подается именно список целых чисел (integers), а не строка.
2. **НИББД:** Поскольку $S15–S22$ — это уникальный номер из базы НИББД, вам может понадобиться отдельный сервис для получения этого идентификатора клиента перед генерацией счета.

### Добавление функции валидации

Ниже обновленный модуль. Я добавил функцию `valid_account?`, которая принимает МФО и 20-значный счет, собирает их в нужную последовательность и проверяет контрольный ключ.

Elixir

```elixir
defmodule Elixir4ABS.AccountValidator do
  @moduledoc """
  Модуль для валидации и расчета контрольных ключей банковских счетов Узбекистана
  согласно алгоритму клиринга ЦБ.
  """

  @doc """
  Проверяет корректность 20-значного счета.
  Принимает mfo (5 цифр) и account (20 цифр) в виде строк или списков.
  """
  def valid_account?(mfo, account) when is_binary(mfo) and is_binary(account) do
    mfo_digits = String.graphemes(mfo) |> Enum.map(&String.to_integer/1)
    acc_digits = String.graphemes(account) |> Enum.map(&String.to_integer/1)
    
    valid_account?(mfo_digits, acc_digits)
  end

  def valid_account?(mfo_digits, acc_digits) when length(mfo_digits) == 5 and length(acc_digits) == 20 do
    # Извлекаем текущий ключ из 9-й позиции счета (индекс 8)
    provided_key = Enum.at(acc_digits, 8)

    # Формируем последовательность для расчета (B1-B5, C6-C10, V11-V13, S15-S22, N23-N25)
    # Исключаем сам ключ (K14) из входных данных для вычисления
    {part1, [_k | part2]} = Enum.split(acc_digits, 8)
    full_sequence = mfo_digits ++ part1 ++ part2

    # Считаем, какой ключ ДОЛЖЕН быть
    calculated_key = calculate_k14(full_sequence)

    provided_key == calculated_key
  end

  @doc """
  Реализация алгоритма из документа ЦБ.
  Принимает 24 цифры (без ключа K14) и возвращает расчетный ключ.
  """
  def calculate_k14(digits) when length(digits) == 24 do
    # Сумма произведений соседних пар: (B1*B2) + (B2*B3) ... + (N24*N25)
    pairs_sum = 
      digits
      |> Enum.chunk_every(2, 1, :discard)
      |> Enum.map(fn [a, b] -> a * b end)
      |> Enum.sum()

    # Финальное слагаемое (N25 * 9)
    total_sum = pairs_sum + (List.last(digits) * 9)

    remainder = rem(total_sum, 11)

    # Логика исключений согласно документу
    case remainder do
      0 -> 9
      1 -> 0
      _ -> 11 - remainder
    end
  end
end
```

### Основные моменты реализации:

- **Pattern Matching:** Функция `valid_account?` поддерживает как строки (удобно для API/форм), так и списки цифр.
- **Обработка K14:** Обратите внимание, что в `calculate_k14` теперь передается 24 цифры. Программа сама «вырезает» 9-й разряд из номера счета, чтобы проверить его валидность.
- **Исключения:** Логика `0 -> 9` и `1 -> 0` реализована через `case` для наглядности.

### Пример использования:

Elixir

```
# Проверка счета
is_ok = Elixir4ABS.AccountValidator.valid_account?("00444", "20208000112345678001")
IO.puts("Счет корректен: #{is_ok}")
```

---
### Почему это важно?

Если вы ошибетесь хотя бы в одной цифре номера или МФО, контрольная сумма не совпадет, и банковская система выдаст ошибку «Неверный контрольный ключ». Это защищает от отправки денег «в никуда».
